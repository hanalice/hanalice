## 🎨 1. 为什么选择“数据驱动的 README”？

在 GitHub 上，我们的 Profile README 是个人的技术门面。但如果每次写完新博客，都要手动去修改 `README.md` 中的文章列表、标签云，不仅极其繁琐，还容易产生格式错乱。

因此，本项目采用了 **“单一数据源 (Single Source of Truth)”** 的设计模式：
- **只编辑模板**：我们只维护结构固定的 `README.template.md`，它包含主页的静态介绍以及若干个特殊的 HTML 注释占位符（例如 `<!-- BLOG-POST-LIST:START -->`）。
- **自动化编译**：通过后台脚本读取 `/posts` 目录下的所有文章，并自动将数据渲染、注入到占位符中，最终生成不可手动编辑的 `README.md`。

这种“数据与表现分离”的理念，是本系统能够 100% 自动化的工程基石。

## 🛠️ 2. 自动化体系的三大支柱

整个自动化运维工作流由三个专职的 GitHub Actions 共同承载：

### 支柱一：定时与事件双触发的同步核心 (`blog-sync.yml`)
为了让门户主页保持源源不断的生命力，构建工作流在两种场景下会被自动唤醒：
1. **主动触发**：只要我们在本地写完文章，用 Git 推送（Push）到 `posts/` 目录时，工作流会立刻识别并启动，在数秒内更新主页。
2. **被动轮换**：系统配置了每日凌晨零点（UTC）执行的 `cron` 定时器。即使我们没有写新博文，定时器也会唤醒系统，重新轮换今日的像素小人（Pixel Mascot），保证访问者每天看到的都是不一样的小人。

### 支柱二：防止死循环的图片压缩 (`image-compress.yml`)
在极简设计中，图片加载速度至关重要。我们引入了自动压缩图片的 Actions 流程。然而在工程实践中，有一个经典的**死循环黑洞**：
> 压缩图片 Action 执行完毕后，需要自动将压缩后的图片 commit 并 push 回仓库。但这会触发新的 push 事件，从而再次唤醒图片压缩流，导致无限死循环。

**我们的解决方案**：在自动 commit 时，显式地在 commit 消息中附带 `[skip ci]` 标记。GitHub 在收到该标记后会自动阻断后续的工作流触发，优雅地打破了无限循环链。

### 支柱三：自动巡检的死链检测 (`link-checker.yml`)
互联网上的链接随时可能失效。为了防止主页上的外部参考链接变成 404，每月 1 号系统会自动运行一次链接健康检查。一旦发现死链，它不会粗暴地让构建挂掉，而是会在仓库中自动提起一个 Issue 提醒我们去手动修复。这种“非阻断式警报”设计，大大提升了系统运行的鲁棒性。

## 🧩 3. 趣味点睛：无 Git 膨胀的每日像素小人轮换

主页顶部每天自动轮换的 Apple 极简风格像素小人是本项目的趣味担当。

从工程角度来看，如果每次轮换都去覆写、commit 一个新的二进制图片文件，会导致 Git 历史中堆积大量无意义的二进制差异，使 `.git` 文件夹体积迅速膨胀。

为了解决这个问题，我们的 Python 脚本在运行 `rotate_mascot()` 时采取了巧妙的双重设计：
- **物理文件安全隔离**：在 `assets/mascots/` 目录下预置 10 张固定好的小人图片（`01.png` ~ `10.png`）。
- **动态链接注入**：脚本通过替换模板中的 `{{DAILY_MASCOT}}` 占位符，动态指向当前抽中的小人文件名。只修改文本而不修改物理大图文件，使 Git 提交仅产生极小的文本改动，完美杜绝了仓库历史膨胀。
